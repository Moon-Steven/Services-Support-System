# Around the Clock 模块 — 产品需求文档 (PRD)

> 版本：v1.0 · 最后更新：2026-04-07
> 归属：Around the Clock 产品线
> 状态：V1 已交付（前端 Mock），V2 待开发

---

## 1. Overview

### 产品定义

Around the Clock 是 **AI 代理执行力与策略透明度的可视化中枢**，同时也是**业务流转与内容审核网关**。

### 问题陈述

HUI 系统 7×24 小时执行广告投放策略，每日产生大量策略事件（出价调整、素材淘汰、预算平滑、异常拦截等）。这些事件在对客展示前存在三层障碍：

1. **安全障碍**：策略事件包含 P0 级算法细节（如 PacingRatio、出价系数），直接暴露给客户存在商业风险
2. **可读性障碍**：原始事件为碎片化技术数据，客户无法理解"AI 在帮我做什么"
3. **时效障碍**：策略事件必须在 4 小时内完成审核并推送至客户前台，否则失去"Always On"的即时感

### 解决方案

Around the Clock 由两个核心子模块构成：

| 子模块 | 解决的障碍 | 作用 |
|--------|-----------|------|
| **Timeline 配置** | 可读性障碍 | 为每个客户配置 24h 内容模板、语气风格和模糊化强度 |
| **审核工作台** | 安全 + 时效障碍 | 三级角色审核流水线 + SLA 时效管控，确保内容安全且及时 |

### 核心目标

| 阶段 | 目标 |
|------|------|
| V1（当前） | 完成前端 Mock + 审核流转 UI，在内部系统跑通展示和交互流程 |
| V2 | 接入 HUI 系统真实事件源，打通全链路审核发布 |
| V3 | 对客 Timeline 面板上线，实现策略事件的实时可视化展示 |

### 目标用户

| 角色 | 使用场景 | 阶段 |
|------|----------|------|
| **交付团队** | 事实核查：验证 AI 动作描述是否准确，纠正事实性偏差 | V1 |
| **行业运营** | 业务包装：润色话术，注入行业语境和"勤勉度"表达 | V1 |
| **销售** | 客情把控：终审发布或拦截，判断对客时机和措辞 | V1 |
| **广告主/客户** | 在前台 Timeline 面板查看 AI 24h 策略动态 | **V3** |

### 范围

**V1 包含：**

- 客户级 Timeline 配置（时间点 + 内容模板 + 语气风格 + 模糊化强度）
- 三级审核工作台（交付 → 运营 → 销售），含角色切换
- SLA 交通灯机制（红/黄/绿 + 时效统计卡片）
- 18 条 mock 审核事件数据，覆盖多客户多阶段
- 叙事链结构（信号 → 策略 → 动作 → 结果）
- 操作时间轴 / 审计日志
- Diff 高亮（文案修改前后对比）
- 批量操作（批量通过 / 批量驳回）
- Timeline 配置 ↔ 审核工作台 双向跳转关联

**V1 明确不包含：**

- HUI 系统真实事件接入（V2）
- 超时自动降级/兜底机制的后端实现（V2）
- 对客前台 Timeline 面板（V3）
- 企微/钉钉消息通知（V2 增强）
- 实验追踪进度条 UI（V2）

## 2. Background & Context

### 为什么是现在

1. **HUI 系统已稳定运行** — 每日产生策略事件，但缺乏面向客户的呈现层
2. **内部反馈** — 销售团队反馈"缺乏工具向客户证明 AI 一直在工作"，依赖口头描述
3. **客户信任瓶颈** — 客户对凌晨时段的自动化操作（调价、淘汰素材）接受度偏低

### 产品逻辑链

```
Around the Clock（AI 做了什么）→ Persona（AI 是谁）→ Data Results（效果如何）
```

Around the Clock 审核通过的正面策略事件将反馈至 Persona 雷达的维度分数，形成信任循环。

## 3. User Stories & Requirements

### P0 — Must Have（V1 已交付）

#### US-01：按客户配置 24h 策略内容模板

> **As a** 交付团队成员，**I want to** 为每个客户配置时间点+内容的 Timeline 模板，**so that** AI 策略事件能按预设模板转化为可读文案。

**验收标准：**

- 每个客户独立配置，支持从行业模板初始化
- 每个条目包含：时间点、事件类型、叙事链（信号/策略/动作/结果）、对客文案
- 支持添加/编辑/删除/启用/禁用条目
- 支持 3 种语气风格切换（专业版/幽默版/轻松版）
- 支持 3 级客户认知水平设定（入门/标准/专业），影响模糊化强度
- 时钟预览可视化展示 24h 分布
- 支持发布配置到客户端

#### US-02：三级角色审核流水线

> **As a** 运营负责人，**I want to** 策略事件经过交付→运营→销售三级审核后才对客发布，**so that** 确保内容的准确性、专业度和客情适配性。

**验收标准：**

- 交付团队（一审）：提交运营 / 修改并提交 / 废弃拦截
- 行业运营（二审）：提交销售 / 话术润色 / 退回交付
- 销售（三审）：确认发布 / 最终微调 / 隐藏拦截 / 撤回已发布
- 角色切换后显示该角色的待办事件
- 每级操作记录审计日志，下游可见上游修改痕迹

#### US-03：SLA 时效管控

> **As a** 运营主管，**I want to** 每个事件有明确的处理时限和超时预警，**so that** 确保 4 小时内完成审核推送。

**验收标准：**

- 三色交通灯：绿色（充裕 >1h）、黄色（临近 <1h）、红色（已超时）
- 顶部 SLA 统计卡片，显示各状态数量，可点击筛选
- 统计卡片包含语义图标（超时⊘ / 临近◔ / 充足◉）
- SLA 筛选与下拉筛选器联动

#### US-04：文案编辑与 Diff 追踪

> **As a** 行业运营，**I want to** 在审核时编辑文案并查看修改痕迹，**so that** 上下游能清晰看到每一步的修改内容。

**验收标准：**

- 详情 Drawer 内可编辑对客文案
- 审计时间轴记录每次编辑（含操作人、时间、修改内容快照）
- 支持查看修改前后对比（Diff 高亮）

#### US-05：批量操作

> **As a** 销售，**I want to** 批量选择同类事件一键通过或驳回，**so that** 提升日常审核效率。

**验收标准：**

- 列表支持复选框多选
- 选中后显示已选数量，激活批量操作按钮
- 支持批量通过（提交下一环节）和批量驳回
- 操作后 Toast 反馈

#### US-06：Timeline 配置与审核工作台双向关联

> **As a** 交付团队成员，**I want to** 在客户 Timeline 配置中看到该客户的待审事件数，点击可直接跳转审核，**so that** 配置和审核两个工作流无缝衔接。

**验收标准：**

- 客户卡片显示待审事件数（如 "● 2 待审"），点击跳转审核工作台并自动筛选该客户
- 审核工作台选中客户时显示上下文条（头像 + 名称 + 行业 + "的审核事件"）
- 上下文条提供 "查看 Timeline 配置" 快捷链接
- 上下文条提供 "查看全部" 清除筛选
- Tab 徽标显示全局总待审数

### P0 — Must Have（V2 待开发）

#### US-07：HUI 真实事件接入

> **As a** 系统架构师，**I want to** 将 HUI 系统的策略事件实时接入审核工作台，**so that** 从 Mock 数据过渡到真实数据。

**验收标准：**

- 对接 HUI 事件推送接口，自动创建审核事件
- 事件包含完整 HUI JSON 参数摘要
- 系统预生成初始文案（基于 Timeline 模板 + 模糊化引擎）
- 自动计算 SLA 截止时间（事件发生 + 4h）

#### US-08：超时兜底机制

> **As a** 运营主管，**I want to** P2 事件超时后自动降级处理，**so that** 避免事件永久卡在流水线中。

**验收标准：**

- P2 事件在运营环节超时 → 沿用默认文案跳至销售
- 销售环节超时 12h → 自动转为隐藏拦截
- 超时操作写入审计日志（actor = "system"）
- 高敏感度事件（P0/P1）不自动降级，仅升级通知主管

### P1 — Should Have

#### US-09：实验追踪进度条

> **As a** 交付团队成员，**I want to** 为 E1-E7 类实验事件显示进度条，**so that** 可视化从"数据积累中"到"验证成功"的全过程。

#### US-10：消息通知

> **As a** 审核人，**I want to** 有新待办或事件超时时收到通知，**so that** 不需要反复刷新工作台。

**验收标准：**

- V2 站内待办红点 + 待办计数
- V2 增强：企微/钉钉 Bot 通知

### P2 — Nice to Have

#### US-11：前台 Timeline 面板

> **As a** 广告主/客户，**I want to** 在前台看到 AI 的 24h 策略动态时间线，**so that** 直观了解 AI 一直在帮我优化投放。

#### US-12："勤勉度"可视化增强

> **As a** 销售，**I want to** Timeline 面板标注"非办公时段"的决策点，**so that** 向客户展示 AI 在凌晨3点等时段的活跃决策。

## 4. Design & UX

### 核心流程：事件生成 → 三级审核 → 对客发布

```
┌─────────────────────────────────────────────────────────────┐
│  阶段 0：事件生成（HUI 系统）                                 │
│                                                             │
│  HUI 策略引擎执行动作后自动触发：                               │
│  ① 生成原始策略事件（含 HUI JSON 参数）                        │
│  ② 基于 Timeline 模板 + 模糊化引擎预生成对客文案                 │
│  ③ 评估敏感度等级（P0 高危 / P1 敏感 / P2 常规）                │
│  ④ 计算 SLA 截止时间（occurredAt + 4h）                       │
│                                                             │
│  输出：AtcReviewEvent（currentStage = at_delivery）           │
└──────────────────────────┬──────────────────────────────────┘
                           │
┌──────────────────────────▼──────────────────────────────────┐
│  阶段 1：一审 — 交付团队                                      │
│                                                             │
│  审核内容：                                                   │
│  · HUI 底层参数 vs 系统预生成文案，核查事实准确性                  │
│  · 叙事链（信号→策略→动作→结果）是否逻辑自洽                     │
│                                                             │
│  操作：                                                      │
│  · [提交运营] → at_ops                                       │
│  · [修改并提交] → 修正文案后进入 at_ops                         │
│  · [废弃拦截] → blocked（终止流转）                             │
└──────────────────────────┬──────────────────────────────────┘
                           │ 通过
┌──────────────────────────▼──────────────────────────────────┐
│  阶段 2：二审 — 行业运营                                      │
│                                                             │
│  审核内容：                                                   │
│  · 话术是否体现行业语境和"勤勉度"                               │
│  · 数据脱敏是否到位（P0 算法细节不得泄露）                        │
│                                                             │
│  操作：                                                      │
│  · [提交销售] → at_sales                                     │
│  · [话术润色] → 修改文案后继续流转                               │
│  · [退回交付] → at_delivery（附驳回原因）                       │
└──────────────────────────┬──────────────────────────────────┘
                           │ 通过
┌──────────────────────────▼──────────────────────────────────┐
│  阶段 3：三审 — 销售                                          │
│                                                             │
│  审核内容：                                                   │
│  · 结合当前客情判断发布时机                                      │
│  · 前台预览效果确认                                            │
│                                                             │
│  操作：                                                      │
│  · [发布] → published（同步至客户前台）                          │
│  · [最终微调] → 修改后直接发布                                   │
│  · [隐藏拦截] → blocked（客情不佳，扣留不发）                     │
│  · [撤回] → blocked（已发布内容下线）                            │
└─────────────────────────────────────────────────────────────┘
```

### 叙事链结构

每个策略事件遵循 **信号 → 策略 → 动作 → 结果** 的因果链路：

| 环节 | 说明 | 示例 |
|------|------|------|
| 信号 | 触发 AI 行动的数据信号 | "凌晨时段监测到转化波动" |
| 策略 | AI 的推理/判断 | "AI 已按风控阈值暂停异常素材" |
| 动作 | AI 执行的具体操作 | "预计为您守住夜间预算安全" |
| 结果 | 预期或已达成的业务价值 | "成功节省 ¥8.2k 预算" |

### 动态模糊化引擎

根据客户认知水平（`client_sophistication`）渲染不同粒度的文案：

| 认知水平 | 模糊化强度 | 示例 |
|----------|-----------|------|
| 入门 `basic` | 强 | "AI 智能预算管理已介入，获客成本表现优秀" |
| 标准 `standard` | 中 | "AI 已按风控阈值暂停异常素材，预计节省 ¥8k+" |
| 专业 `advanced` | 弱 | "PacingRatio 触发风控阈值 0.85，已暂停素材 #1234，CPA 降幅 12%" |

### SLA 交通灯

| 颜色 | 条件 | 含义 |
|------|------|------|
| 🟢 绿色 | 剩余 > 1h | 时效充足 |
| 🟡 黄色 | 剩余 < 1h | 临近超时 |
| 🔴 红色 | 已超时 | 需紧急处理 |

### 敏感度分级

| 等级 | 含义 | 超时策略 |
|------|------|----------|
| **P0 高危** | 包含核心算法参数 | 不自动降级，升级通知主管 |
| **P1 敏感** | 包含出价/预算调整细节 | 不自动降级，升级通知主管 |
| **P2 常规** | 常规策略动态 | 运营超时 → 沿用默认文案跳销售；销售超时 12h → 自动隐藏 |

### Edge Cases

| 场景 | 处理 |
|------|------|
| P2 事件运营环节超时 | 沿用默认文案自动跳至销售（actor = "system"） |
| 销售环节超时 12h | 自动转为隐藏拦截，避免过期信息触达客户 |
| P0/P1 事件超时 | 不自动降级，升级通知至主管 |
| 销售撤回已发布内容 | 标记为 blocked，前台不再展示 |
| 预算变动 <1% | 交付可直接废弃拦截，标记为无效/琐碎动作 |
| 上下游文案修改冲突 | 后写覆盖前写，审计日志保留完整修改历史 |

## 5. Technical Considerations

### 数据模型

#### AtcReviewEvent（审核事件实体）

```typescript
type AtcReviewEvent = {
  id: string
  clientId: string
  clientName: string
  accountLabel?: string
  occurredAt: string              // ISO 事件发生时间
  slaDueAt: string                // ISO SLA 截止时间
  sensitivity: 'P0' | 'P1' | 'P2'
  eventType: 'bidding' | 'creative' | 'monitor' | 'strategy' | 'experiment'
  huiPayloadSummary: string       // HUI 原始参数摘要
  customerCopyDraft: string       // 对客文案（可编辑）
  previousCustomerCopy?: string   // 上一版本文案（用于 Diff）
  currentStage: AtcPipelineStage  // at_delivery | at_ops | at_sales | published | blocked
  blockedReason?: string
  narrative: {
    signal: string
    strategy: string
    action: string
    outcome: string
  }
  kpiRefs: ('B1' | 'C4' | 'M6')[]
  experimentProgress?: {
    label: string
    currentStep: number
    totalSteps: number
  }
  auditLog: AtcAuditLogEntry[]
}
```

#### AtcAuditLogEntry（审计日志条目）

```typescript
type AtcAuditLogEntry = {
  id: string
  at: string                      // ISO 操作时间
  action: AtcAuditAction          // 操作类型枚举
  actor: string                   // 操作人（系统操作为 "system"）
  note?: string                   // 审核意见
  copySnapshot?: string           // 文案快照（用于 Diff 对比）
}
```

#### AtcAuditAction（操作类型枚举）

```typescript
type AtcAuditAction =
  | 'hui_triggered'               // HUI 系统触发
  | 'delivery_submit_ops'         // 交付提交运营
  | 'delivery_edit'               // 交付修改文案
  | 'delivery_reject'             // 交付废弃拦截
  | 'ops_submit_sales'            // 运营提交销售
  | 'ops_polish'                  // 运营润色话术
  | 'ops_return_delivery'         // 运营退回交付
  | 'sales_publish'               // 销售发布
  | 'sales_edit'                  // 销售微调
  | 'sales_hide'                  // 销售隐藏拦截
  | 'sales_recall'                // 销售撤回已发布
  | 'system_timeout_forward'      // 系统超时自动流转
  | 'system_timeout_hide'         // 系统超时自动隐藏
```

### KPI 指标映射

| KPI 编码 | 指标 | Around the Clock 中的闭环展示 |
|----------|------|------|
| B1 | 预算平滑 | 展示日预算从 {old} 调至 {new} 如何确保平稳消耗 |
| C4 | 素材迭代 | 展示 AI 基于诊断结果提出的迭代建议及新素材生成 |
| M6 | 健康评分 | 全天候动作后的项目整体健康度分值变化 |

### 文件结构

| 文件 | 职责 |
|------|------|
| `lib/around-the-clock.ts` | 类型定义、常量、SLA 计算、Mock 数据种子 |
| `lib/data.ts` | 客户数据、Timeline 模板、行业模板 |
| `app/clock-config/page.tsx` | Timeline 配置 + 审核工作台（双 Tab 主页面） |
| `app/around-the-clock/review/page.tsx` | 审核工作台独立页面（已合并入 clock-config） |
| `components/layout/Sidebar.tsx` | 侧边栏导航（Around the Clock 入口） |

### 性能

- SLA 交通灯计算为 O(1) 时间比较，无性能瓶颈
- 事件列表为内存筛选，当前 Mock 18 条，生产需后端分页
- 客户卡片的待审数计算为 O(N×M) 遍历，N=客户数 M=事件数，当前规模可接受
- 详情 Drawer 为单例，避免重复渲染

### 安全 & 隐私

- P0 级算法参数（PacingRatio 等）不得在对客文案中出现
- 模糊化引擎根据客户认知水平自动脱敏
- 所有审核操作通过 AuditLog 记录完整审计链
- HUI 参数摘要仅内部可见，不对客展示
