# Persona 模块 — 产品需求文档 (PRD)

> 版本：v1.1 · 最后更新：2026-04-07
> 归属：Around the Clock 产品线
> 变更记录：v1.1 修复问题陈述逻辑、重新校准优先级与阶段归属、补充缺失角色与灰度方案

---

## 1. Overview

### 问题陈述

客户在使用 AI Agent 管理广告投放时，存在两层障碍：

1. **理解障碍**：AI Agent 的 6 个能力维度（留存、探索、精准、效率、风控、创意）缺乏可视化呈现，客户无法判断"AI 在哪些方面帮了我"
2. **情感障碍**：即使理解了能力数据，面对纯数字的雷达图，客户仍然感到"在和机器打交道"，缺乏拟人化的信任锚点

这两层障碍导致客户对高价值自动化操作（如凌晨自动调价、素材淘汰）的接受度偏低。

### 解决方案与假设

Persona 模块由两个**独立但互补**的组件构成，各自解决一层障碍：


| 组件       | 解决的障碍 | 作用                           | 优先级      |
| -------- | ----- | ---------------------------- | -------- |
| **能力雷达** | 理解障碍  | 可视化 6 维能力 + 历史趋势，回答"AI 擅长什么" | **核心**   |
| **名人名言** | 情感障碍  | 赋予 Agent 可感知的价值观人格，降低"机器感"   | **辅助增强** |


> **关键假设 [待验证]**：名言作为情感层包装，能够提升客户对雷达数据的关注度和正面解读意愿。此假设需在 V1 内部试用阶段收集定性反馈验证。如果验证失败，名言可独立移除而不影响雷达的核心价值。

### 目标

**V1 目标（内部验证）**：完成数据模型 + 前端组件 + 种子数据，在内部系统中跑通展示流程，收集交付/销售团队的定性反馈。

**V2 目标（灰度上线）**：接入真实数据源，对 20% 客户灰度展示，验证信任度假设。

### 目标用户


| 角色                 | 使用场景                                 | 阶段         |
| ------------------ | ------------------------------------ | ---------- |
| **交付团队**           | 在内部客户详情页查看 Persona，评估是否准确反映 Agent 行为 | V1         |
| **销售**             | 使用 Persona 作为客情沟通工具，向客户解读 AI 能力      | V1         |
| **行业运营**           | 审核名言的行业适配性和品牌安全性                     | V2         |
| **名言内容策划人**（品牌/运营） | 发现、录入、标注名言，维护名言库的供给侧                 | V2         |
| **算法/数据团队**        | 定义雷达维度的计算公式，标注名言亲和力权重                | V2         |
| **广告主 / 客户**       | 在客户端查看 Persona，理解 AI Agent 的特征       | **V2 灰度后** |


### 范围

**V1（内部验证版）包含：**

- 6 维能力雷达可视化（含历史对比）
- 名言库种子数据（10 条，预审核状态）
- 名言与雷达的匹配算法
- 客户详情页（内部系统）侧栏集成
- 数据模型含锁定字段（为 V2 审核流程预留）

**V1 明确不包含：**

- 对客展示（V2 灰度后开放）
- 审核/锁定操作 UI（V2）
- 雷达分数的真实计算引擎（V2，V1 使用策略 mock 并标注"演示数据"）
- 名言库管理后台 CMS（V3）

> **⚠️ V1 数据诚实度约束**：V1 的雷达分数为策略团队基于经验设定的演示数据，非 Agent 行为真实计算结果。所有 V1 界面必须标注"演示数据"标记，禁止以任何形式暗示这是实时计算结果。

## 2. Background & Context

### 为什么是现在

1. **Around the Clock 已上线** — 正面策略事件的审核发布流程已具备，Persona 作为"AI 做了什么"到"AI 是谁"的自然延伸，基础设施就绪
2. **内部反馈 [定性]** — 销售团队在客情沟通中多次反馈"缺乏直观工具向客户解释 AI 在做什么"，目前依赖口头描述，效率低且不一致
3. **行业趋势 [观察]** — 广告技术行业开始出现 Agent 能力透视类功能（如 Google Ads 的"自动出价策略强度"指示器），方向值得探索

> **注**：以上第 2、3 点为团队内部观察，非量化数据。V1 的一个目标是通过内部试用将这些定性判断转化为可量化的证据。

### 与现有系统的关系

```
┌───────────────────────────────────────────────────┐
│                 客户详情页（内部系统）               │
│  ┌──────────┐  ┌────────────────────────────────┐ │
│  │ Persona  │  │   Data Results & AI Decisions  │ │
│  │ · 雷达   │  │   · KPI 指标卡片                │ │
│  │ · 名言   │  │   · 趋势图表                    │ │
│  │          │  │   · 策略决策卡片                │ │
│  └──────────┘  ├────────────────────────────────┤ │
│                │   Around the Clock             │ │
│                │   · 24h Timeline               │ │
│                │   · 审核工作台                   │ │
│                └────────────────────────────────┘ │
└───────────────────────────────────────────────────┘
```

**产品逻辑链：**

```
Persona（AI 是谁）→ Around the Clock（AI 做了什么）→ Data Results（效果如何）
```

V2 实现后，正面策略事件经审核发布将反馈至 Persona 雷达的维度分数，形成正向循环。V1 阶段此循环暂未打通。

### 关键指标基线


| 指标              | 当前值 | 目标                  | 说明           |
| --------------- | --- | ------------------- | ------------ |
| 客户对 AI 决策的信任度   | 无基线 | V1 内部定性评估；V2 建立量化基线 | 需定义测量方法      |
| 销售解释 AI 能力的沟通效率 | 无基线 | V1 收集交付/销售主观反馈      | 定性 → 定量      |
| Persona 卡片关注度   | N/A | V2 灰度阶段建立           | 需定义"关注"的埋点口径 |

## 3. User Stories & Requirements

### P0 — Must Have（V1 交付）

#### US-01：内部用户查看 AI Agent 能力画像

> **As a** 交付团队成员/销售，**I want to** 在客户详情页看到该客户 AI Agent 的能力雷达和特征名言，**so that** 我能向客户准确解释 AI 在管理其投放时的能力分布。

**验收标准：**

- 6 维雷达图展示当前分数（Retention / Exploration / Precision / Efficiency / Risk Ctrl / Creativity）
- 叠加展示 7 天前分数作为对比层（灰色虚线）
- 显示评估周期日期范围
- 显示匹配的名人名言（含作者、年份）
- 雷达下方分数表显示每维度的变化 delta
- **界面显著标注"演示数据"**（V1 约束）

#### US-02：名言库种子数据预审

> **As a** 产品负责人，**I want to** 名言库的种子数据在入库时已经过品牌安全预审，**so that** V1 内部试用期间不会出现不当内容。

**V1 验收标准（简化版）：**

- 种子名言在代码入库前由品牌/运营负责人口头确认适用性
- 名言数据结构包含 `status` 字段，V1 种子全部标记为 `available`
- 数据结构预留 `reviewedBy` / `reviewedAt`，V1 填写预审信息
- 数据结构预留 `retiredReason`，V1 暂不使用

> 注：完整的名言入库审核 UI（draft → under_review → available → retired 全流程）在 V2 交付。V1 通过代码审查 + 口头确认替代。

#### US-03：名言锁定数据模型预留

> **As a** 系统架构师，**I want to** V1 的数据模型包含锁定状态字段，**so that** V2 构建审核 UI 时无需重构数据层。

**V1 验收标准（数据层 only）：**

- `PersonaSnapshot` 包含 `lockStatus` 字段（`auto_selected / in_review / locked / unlocked`）
- `canAlgorithmReplaceQuote()` 函数实现锁定保护逻辑
- V1 种子数据中包含各种 `lockStatus` 示例以验证数据结构
- **V1 无操作 UI**，锁定状态仅通过种子数据演示

### P0 — Must Have（V2 交付）

#### US-04：名言库完整生命周期管理

> **As a** 名言内容策划人，**I want to** 通过审核流程录入和管理名言，**so that** 只有通过合规审核的名言才能进入可选池。

**验收标准：**

- 名言状态生命周期 UI：`draft → under_review → available → retired`
- 只有 `status === 'available'` 的名言可被匹配算法选中
- 下架分级：`retired_normal`（自然过期）和 `retired_safety`（安全原因）
- `retired_safety` 触发强制解锁（见 US-06）

#### US-05：审核通过的名言不被自动替换

> **As a** 销售，**I want to** 确保审核通过的 Persona 名言不会被算法自动换掉，**so that** 我向客户解读过的内容保持一致，不影响客情。

**验收标准：**

- `locked` 状态下，匹配算法不可自动替换
- 只有人工操作才能将 `locked` 变为 `unlocked`，需填写原因
- 重新匹配后回到 `auto_selected`，需再次走完审核
- UI 提供锁定/解锁操作入口

#### US-06：安全下架强制解锁

> **As a** 品牌合规审核员，**I want to** 因安全原因下架名言时自动解除所有引用该名言的客户锁定，**so that** 有风险的内容不会因为锁定机制而继续展示。

**验收标准：**

- `retired_safety` 操作触发：遍历所有引用该名言的 `PersonaSnapshot`
- 将相关 snapshot 的 `lockStatus` 强制改为 `unlocked`
- 写入系统审计日志，记录原因和影响范围
- 触发匹配算法为受影响客户重新推荐名言

### P1 — Should Have

#### US-07：名言与雷达的智能匹配

> **As a** 交付团队成员，**I want to** 系统基于客户 AI Agent 的能力雷达自动推荐最匹配的名言，**so that** 名言与 Agent 实际表现保持一致，减少人工选择成本。

**验收标准：**

- 基于余弦相似度计算名言亲和力向量与客户雷达向量的匹配度
- 行业标签匹配加分（同行业 +0.05 bonus）
- 输出匹配分数 `quoteMatchScore`（0–1）
- 仅从 `available` 状态的名言池中选择
- 匹配分数 < 0.5 时不自动绑定，标记"待人工选择"

#### US-08：Persona 状态标签展示

> **As a** 交付团队成员/销售，**I want to** 在客户详情页看到 Persona 的锁定状态标签，**so that** 我知道当前名言是否已审核锁定、是否可以向客户展示。

**验收标准：**

- `locked` → 青色标签"已锁定"
- `in_review` → 橙色标签"审核中"
- `auto_selected` → 灰色标签"算法推荐"
- `unlocked` → 灰色标签"待重新匹配"

#### US-09：客户绑定业务审核

> **As a** 交付团队成员，**I want to** 对算法推荐的名言-客户绑定进行业务审核，**so that** 确认名言适合该客户的具体情境后再展示。

**验收标准：**

- 复用 Around the Clock 的三级审核架构（交付 → 运营 → 销售）
- 交付审核：核查名言与客户雷达的匹配合理性
- 运营审核：核查名言是否符合该客户所在行业的调性
- 销售审核：最终确认是否适合对客展示、是否影响客情
- SLA 时效管理（复用现有 SLA 交通灯机制）

> **名言库入库审核 vs 客户绑定审核的关系**：这是两个独立流程。入库审核解决"这条名言本身是否安全可用"（合规层，一次性）；绑定审核解决"这条名言是否适合这个客户"（业务层，每次绑定）。一条名言通过入库审核 ≠ 适合所有客户。例如：Buffett 的风险名言通过了合规审核，但绑定到一个追求激进增长的客户上可能不合适。

### P2 — Nice to Have

#### US-10：维度分数剧变告警

> **As a** 交付团队成员，**I want to** 在单维度分数周环比变化超过 15% 时收到提醒，**so that** 我可以核查原因并附加变更说明。

**验收标准：**

- 系统自动检测维度变化幅度
- 超阈值时在 Persona 卡片标注告警图标
- 支持附加变更说明

## 4. Design & UX

### 核心流程：数据生成 → 三级审核 → 对客展示

这是 Persona 模块的**主干流程**。任何 Persona 内容必须走完此流程才可对客展示，无例外。

```
┌─────────────────────────────────────────────────────────────────────┐
│  阶段 0：数据生成（HUI 系统）                                         │
│                                                                     │
│  HUI 系统每个评估周期（7天）自动执行：                                   │
│  ① 计算客户 AI Agent 的 6 维雷达分数（基于 Agent 行为数据）              │
│  ② 从名言可选池（status=available）中匹配最佳名言                       │
│  ③ 生成 PersonaSnapshot（lockStatus=auto_selected）                  │
│                                                                     │
│  输出：雷达分数 + 匹配名言 + 匹配分数                                   │
│  状态：auto_selected（算法推荐，未经人工审核，不可对客展示）                │
└────────────────────────┬────────────────────────────────────────────┘
                         │
                         │ matchScore ≥ 0.5
                         │ （< 0.5 则标记"待人工选择"，不进入审核流水线）
                         │
┌────────────────────────▼────────────────────────────────────────────┐
│  阶段 1：一审 — 交付团队                                              │
│                                                                     │
│  审核内容：                                                          │
│  · 雷达分数是否合理反映 Agent 近期行为？（对照 Around the Clock 事件）    │
│  · 名言文本是否通顺、无歧义？                                          │
│  · 分数变化幅度是否需要附加说明？（单维度变化 > 15% 需标注）               │
│                                                                     │
│  操作：通过 → 进入二审 ｜ 驳回 → 退回 HUI 重新生成 / 人工调整            │
│  状态变更：auto_selected → in_review                                  │
└────────────────────────┬────────────────────────────────────────────┘
                         │ 通过
┌────────────────────────▼────────────────────────────────────────────┐
│  阶段 2：二审 — 行业运营（行运）                                       │
│                                                                     │
│  审核内容：                                                          │
│  · 名言归属人是否存在品牌安全风险？                                      │
│  · 名言调性是否符合该客户所在行业？                                      │
│  · 名言与雷达的匹配是否合理？（如风控型客户配了创意型名言 → 不合适）         │
│                                                                     │
│  操作：通过 → 进入三审 ｜ 驳回 → 退回交付 ｜ 润色文案 → 继续流转          │
└────────────────────────┬────────────────────────────────────────────┘
                         │ 通过
┌────────────────────────▼────────────────────────────────────────────┐
│  阶段 3：三审 — 销售                                                  │
│                                                                     │
│  审核内容：                                                          │
│  · 结合客情判断：此内容是否适合当前展示给该客户？                          │
│  · 是否与近期客户沟通中的承诺/说法一致？                                  │
│  · 是否存在过度承诺风险？（如分数全部 > 90）                              │
│                                                                     │
│  操作：通过 → 锁定发布 ｜ 驳回 → 退回行运 ｜ 微调 → 锁定发布             │
└────────────────────────┬────────────────────────────────────────────┘
                         │ 通过
┌────────────────────────▼────────────────────────────────────────────┐
│  阶段 4：锁定 & 对客展示                                              │
│                                                                     │
│  状态变更：in_review → locked                                        │
│  记录：lockedBy（销售姓名）、lockedAt（时间戳）                         │
│                                                                     │
│  此刻起：                                                            │
│  · Persona 卡片在客户界面生效展示                                      │
│  · HUI 下一周期生成新数据时，算法不可自动替换已锁定内容                   │
│  · 如需更新，必须由销售人工解锁（填写原因）→ 回到阶段 0 重新生成           │
└─────────────────────────────────────────────────────────────────────┘
```

        └──────────────┘

> `**pending_manual` 操作规则**：匹配分数 < 0.5 时系统不自动绑定名言，进入 `pending_manual`。此状态下审核工作台显示"待人工选择"，任何有"手动换名言"权限的角色（交付/行运/销售）均可操作。手动指定后，`lockStatus` 变为 `auto_selected`，`quoteMatchScore` 按新名言重新计算，正常进入三级审核流水线。

**驳回路径的精确语义：**


| 驳回场景    | 发起人 | Snapshot 状态变更                                  | 显示标记                   | 后续处理                              |
| ------- | --- | ---------------------------------------------- | ---------------------- | --------------------------------- |
| 交付驳回    | 交付  | `in_review(at_delivery)` → `auto_selected`     | 标记"退回任务" + 附带驳回原因      | 可人工调整分数/换名言后重新提交，或等待 HUI 下一周期重新生成 |
| 行运驳回→交付 | 行运  | `in_review(at_ops)` → `in_review(at_delivery)` | 标记"退回任务：行运退回" + 附带驳回原因 | 交付可修改后重新提交行运                      |
| 销售驳回→行运 | 销售  | `in_review(at_sales)` → `in_review(at_ops)`    | 标记"退回任务：销售退回" + 附带驳回原因 | 行运可修改后重新提交销售                      |


> **原则**：驳回只退一级，不跨级退回。每次驳回必须附带原因，原因对下游可见。

#### 审核时效（SLA）

复用 Around the Clock 的 SLA 交通灯机制：


| 审核阶段   | SLA 时限 | 超时处理                           |
| ------ | ------ | ------------------------------ |
| 一审（交付） | 24h    | 黄灯告警 → 48h **升级通知**至交付主管，不自动流转 |
| 二审（行运） | 24h    | 黄灯告警 → 48h **升级通知**至运营主管，不自动流转 |
| 三审（销售） | 48h    | 黄灯告警 → 72h 自动拦截（不展示），标记待处理     |


> **不自动流转的理由**：一审核查"分数是否真实反映 Agent 行为"，二审核查"品牌安全"。跳过任何一级等于在未校验的基础上做后续审核，风险不可接受。超时应升级告警而非默认"不审也行"。

#### 人工解锁 → 重新审核

```
locked（已对客展示）
   │
   │ 销售人工解锁（必填原因）
   │
   ▼
unlocked → HUI 重新生成 → auto_selected → 重走阶段 1–4
```

解锁场景举例：

- 客户业务方向变化，需要更换名言调性
- 雷达维度出现较大变化，需更新展示
- 名言归属人出现负面新闻

### 名言库管理流程（独立于客户审核）

名言库的入库审核是**独立流程**，解决"这条名言本身是否安全可用"，与上述客户绑定审核互不依赖。

```
  内容策划人录入 → draft
         │
    合规初审 → under_review
         │
    ┌────┴────┐
  通过      驳回 → 退回 draft
    │
  available（进入可选池，可被 HUI 匹配）
    │
  下架 ──┬── retired_normal（自然过期）
        │     → 已锁定客户保持展示，审核界面标记提示
        │
        └── retired_safety（安全原因）
              → 强制解锁所有引用该名言的客户
              → 触发 HUI 重新匹配 → 重走三级审核
```

> **两个审核的关系**：名言入库审核（合规层，一次性）确保名言本身安全可用。客户绑定审核（业务层，每次绑定）确保该名言适合该客户。一条名言通过入库审核 ≠ 适合所有客户。

| 客户无 Snapshot 数据             | 雷达展示默认值（各维度 70），标注"暂无评估数据"，不显示名言                          |
| 名言因安全原因下架（`retired_safety`） | **强制解锁**所有引用该名言的客户，触发重新匹配，写入审计日志                          |
| 名言因正常原因下架（`retired_normal`） | 已锁定客户保持展示，在审核界面标记"名言已下架"提示运营关注                            |
| 算法匹配分数极低（< 0.5）             | 不自动绑定，进入 `pending_manual` 状态，审核工作台显示"待人工选择"               |
| 多条名言匹配分数相同                  | 优先选择行业标签匹配的，再按审核通过时间排序（优先已验证更久的）                          |
| V1 数据被误认为真实数据               | 所有 V1 界面强制展示"演示数据"标签，不可隐藏                                 |
| 有 `copyOverride` 时展示逻辑      | 对客展示 `copyOverride` 文案，审核 Drawer 同时展示原文和润色版，标注差异          |
| 名言原文被修改（修 typo 等）           | 不自动更新已有 `copyOverride`，发送通知给对应审核人确认是否需更新                  |
| 审核中途换名言                     | `lockStatus` 保持 `in_review` + 当前 `reviewStage` 不变，不重走前序审核 |
| 同一客户连续驳回 3 次                | 第 3 次驳回时系统标记"反复驳回"，升级通知给交付主管                              |

### 匹配算法

1. 将客户 6 维雷达分数归一化为向量
2. 计算每条 `available` 名言亲和力向量与客户向量的**余弦相似度**
3. 行业标签匹配加 0.05 bonus
4. 匹配分数 ≥ 0.5 时取最高分；< 0.5 时不自动绑定

```typescript

  scores: PersonaRadarDimensions,
  industryTag?: string
): { quote: PersonaQuote; matchScore: number }
```

### 状态模型补充说明

`PersonaLockStatus` 枚举中**不包含 `rejected`**。这是有意设计：驳回是一个**瞬时动作**而非持久状态。驳回发生时：

1. `lockStatus` 回退到上一级（交付驳回 → `auto_selected`；行运/销售驳回 → 退回上级的 `in_review`）
2. `rejectedFrom` 记录"从哪一级被退回"（用于 UI 显示"退回任务"标记）
3. `rejectedReason` 记录驳回原因

判断"当前是否为退回任务"的逻辑：

```typescript
function isRejectedTask(snapshot: PersonaSnapshot): boolean {
  return snapshot.rejectedFrom != null && snapshot.rejectedReason != null
}
```

当该 Snapshot 被重新提交审核时，清空 `rejectedFrom` 和 `rejectedReason`。

### 快照保留策略

每个客户会随 HUI 周期性重新生成 Snapshot。历史快照的保留策略：


| 阶段           | 保留策略                                   |
| ------------ | -------------------------------------- |
| V1/V2 (Mock) | 仅保留最近 2 个版本（current + previous），内存存储   |
| V3 (生产)      | 保留最近 12 个版本（约 3 个月），数据库存储 + TTL 自动清理   |
| 审计日志         | 永久保留（合规要求），独立存储在 `PersonaAuditEntry` 表 |


> **数据一致性约束**：当 `copyOverride` 存在时，对客展示应使用 `copyOverride` 而非名言原文。若原名言被编辑（修复 typo 等），`copyOverride` 不自动更新——需由行运/销售重新确认。

### copyOverride 覆写规则

`copyOverride` 用于行运/销售对名言展示文案的润色（如语气调整、措辞优化），遵循以下规则：


| 规则    | 说明                                                                                |
| ----- | --------------------------------------------------------------------------------- |
| 写入时机  | 行运"润色"或销售"微调"操作时写入                                                                |
| 覆盖关系  | **后写覆盖前写**。若行运写了润色版，销售微调后覆盖行运版本                                                   |
| 历史保留  | 每次写入产生一条 `PersonaAuditEntry`（`action = 'copy_override'`），`detail` 字段记录完整的新旧文案，可追溯 |
| 清除时机  | 解锁（`locked → unlocked`）后 HUI 重新生成新 Snapshot 时，`copyOverride` 自动清空（新周期从原文重新审核）     |
| 展示优先级 | 对客展示：`copyOverride > quote.text`。审核 Drawer：同时展示原文和润色版，用 diff 标注差异                 |


### 名言库健康度指标

系统应持续监控名言库状态，在总览页顶部展示告警条：


| 指标       | 阈值                | 告警级别       |
| -------- | ----------------- | ---------- |
| 可用名言总数   | < 20 条            | ⚠ 警告       |
| 单行业可用名言  | < 3 条             | ⚠ 警告       |
| 待审名言堆积   | > 10 条超过 7 天未审    | 🔴 严重      |
| 可用名言被引用率 | > 80% 的名言已被锁定客户引用 | ⚠ 警告（库存紧张） |


### 性能

- 名言匹配为 O(N) 线性遍历，当前库规模 ~10 条，无性能瓶颈
- 当名言库 > 500 条时，应预计算向量并建立行业标签索引
- 雷达 SVG 为纯计算渲染，无运行时依赖
- Snapshot 按 clientId 查询，当前为内存 mock，生产需加索引
- 审计日志写入异步化，不阻塞审核操作

### 安全 & 隐私

- 名言库不含用户数据，无隐私风险
- 雷达分数为 AI 行为聚合指标，不泄露投放细节
- 锁定/解锁操作需记录操作人和时间用于审计
- `retired_safety` 操作需记录完整影响范围
- 所有审核操作通过 `PersonaAuditEntry` 记录完整审计链
- `copyOverride` 修改同样需要审计记录

## 附录 A：名言亲和力标注规则（初版）

> 此规则为 V1 种子数据标注时使用的参考框架，不代表最终标准。V2 需与内容策划人和算法团队共同修订。

### 标注流程

1. 阅读名言，提取核心主题词（如"节约"、"实验"、"数据"、"风险"）
2. 按下表将主题词映射到维度，给出 0–1 的亲和力评分
3. 由第二人独立标注，取两人均值，差异 > 0.3 的维度需讨论对齐

### 主题词 → 维度映射参考


| 主题词          | 主映射维度                 | 次映射维度                 |
| ------------ | --------------------- | --------------------- |
| 节约、成本、ROI、利润 | efficiency (0.7–0.9)  | riskControl (0.3–0.5) |
| 实验、创新、尝试、突破  | exploration (0.7–0.9) | creativity (0.5–0.7)  |
| 数据、测量、证据、精确  | precision (0.7–0.9)   | riskControl (0.3–0.5) |
| 风险、安全、谨慎、防御  | riskControl (0.7–0.9) | retention (0.3–0.5)   |
| 坚持、长期、复利、耐心  | retention (0.7–0.9)   | precision (0.2–0.4)   |
| 创意、洞察、人性、表达  | creativity (0.7–0.9)  | exploration (0.3–0.5) |


### 标注约束

- 每条名言至少有 1 个维度 ≥ 0.7（必须有明确的"强关联"）
- 每条名言至多有 3 个维度 ≥ 0.5（避免"万金油"标注稀释匹配区分度）
- 非相关维度应标注 0.1–0.3（而非 0），保持向量空间的连续性
